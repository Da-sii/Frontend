# 브랜치 관리 방식

## 브랜치 역할

| 브랜치 | 역할 |
|---|---|
| `main` | **빌드/배포가 완료되어 검증된 코드만.** 실제 스토어에 나간(또는 나갈 준비가 끝난) 상태만 반영합니다. |
| `develop` | 개발 통합 브랜치. 기능/수정 작업은 전부 여기로 모입니다. |
| `feat/*`, `fix/*` | 개별 작업 브랜치. `develop`에서 분기해서 작업 후 `develop`으로 PR. |
| `release/{version}` | 버전별 아카이빙 브랜치 (예: `release/1.5.0`). 해당 버전이 배포된 시점의 코드를 그대로 보존. |

## 작업 흐름

```
feat/* , fix/*  →  develop  →  (빌드 완료 후) main
                                   │
                                   └─ release/{version} 브랜치로 아카이빙
```

1. **작업 시작**: `develop`에서 `feat/{설명}` 또는 `fix/{설명}` (이슈 번호가 있으면 `fix/{이슈번호}-{설명}`) 브랜치 생성
2. **PR**: 작업 브랜치 → `develop`으로 PR & 머지
3. **릴리즈 준비**: `develop`에서 빌드/QA까지 끝나서 실제로 내보낼 수 있는 상태가 되면, 그 시점의 코드를 `main`으로 PR & 머지
4. **아카이빙**: `main`에 머지된 그 커밋 기준으로 `release/{version}` 브랜치 생성 (예: `release/1.5.0`)
5. **배포**: `release/{version}` 브랜치(또는 그 시점의 `main`) 위 커밋에 `android-v*`/`ios-v*` 태그를 달아 배포 — 태그 규칙은 [RELEASE.md](./RELEASE.md) 참고

## 브랜치 네이밍

- `feat/{설명}` — 새 기능
- `fix/{설명}` 또는 `fix/{이슈번호}-{설명}` — 버그 수정
- `release/{version}` — 버전 아카이빙 (예: `release/1.5.0`)

> 과거에 쓰던 `update/android-1.0.4` 같은 네이밍은 레거시입니다. 앞으로 버전 아카이빙은 `release/{version}`으로 통일해주세요.

## 규칙

- **`main`에는 빌드가 끝난(=배포 가능한) 상태만 올라갑니다.** 개발 중인 내용은 절대 `main`에 직접 push/머지하지 않습니다.
- 개발 중인 모든 변경은 `develop`을 거칩니다.
- `release/{version}` 브랜치는 해당 버전을 배포한 시점의 스냅샷입니다. 이후 그 버전에 긴급 수정(hotfix)이 필요하면 `release/{version}`에서 분기해서 수정하고, 작업이 끝나면 `main`과 `develop` 양쪽에 반영합니다.
- 작업 브랜치(`feat/*`, `fix/*`)는 `develop`에 머지된 후 삭제해서 브랜치 목록을 깔끔하게 유지합니다.
